Fuzzing

Enumerating resources directly linked to silentium.htb yields nothing. Through subdomain enumeration I manage to retrieve a vhost staging.silentium.htb. A FlowiseAI instance is running on this secondary domain.

k0d14k@kpb:~/Documenti/silentium$ whatweb staging.silentium.htb http://staging.silentium.htb [200 OK] Country[RESERVED][ZZ], HTML5, HTTPServer[Ubuntu Linux][nginx/1.24.0 (Ubuntu)], IP[10.129.23.219], Meta-Author[FlowiseAI], Open-Graph-Protocol[website], Script[module], Title[Flowise - Build AI Agents, Visually], UncommonHeaders[access-control-allow-credentials], nginx[1.24.0]

Not being familiar with the technology, I search for information and discover that version 3.0.4 (released in July 2025) has an RCE tracked as CVE-2025-59528. To correctly use the public exploit provided by Rapid7, two requirements must be met:

  1. being an authenticated user
  2. the software version must actually be that one

I also discover that FlowiseAI exposes an API set at /api/v1. I start a fuzzer to look for useful unauthenticated information. Beyond the version (which turns out to be 3.0.5, ruling out the possible CVE) I find nothing useful.

Going back to the Flowise login page I notice a possible User Enumeration in the password recovery form:

The email must follow the format user@company.com, so in this case I expect [user]@silentium.htb I run an enumeration with Intruder, through which I discover the existence of a user admin with email ben@silentium.htb.

I try to crack the Bcrypt password but none of the wordlists I used managed to give me clear-text credentials.

I therefore went back to investigating known vulnerabilities and found an exploit that chains the execution of CVE-2025-58434 and CVE-2025-59528, available here.

The first one allows, via an Authorization Bypass, to force a password change on an account. The second executes a command once the connection is established.

cat privilege-escalation

Inside ben's session I run sudo -l but cannot run anything as root:

ben@silentium:/tmp$ sudo -l
[sudo] password for ben:
Sorry, user ben may not run sudo on silentium.

I therefore run linpeas.sh again and among the various results I discover the existence of a vhost that had not been detected during enumeration: staging-v2-code.dev.silentium.htb.

Gogs is a self-hosted Git repository. It appears to be affected by several CVEs. I register a user and log in to look for the version but cannot find it.

Searching for known exploits for Gogs I find 3 CVEs. After trying them all in chronological order I discover that the instance is vulnerable to CVE-2025-8110.

After cloning the repository and slightly modifying the script to suit my needs (removing the registration step and replacing the credentials with my own user's) I obtain a shell as root and the root flag.
